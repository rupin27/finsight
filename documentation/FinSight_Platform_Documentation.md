# FinSight Platform Documentation

**Platform snapshot:** July 19, 2026  
**Status:** Current-state technical and product reference  
**Audience:** Developers, product owners, reviewers, operators, and future contributors

> FinSight is a private, multi-currency personal finance intelligence platform. It combines account and transaction management with recurring cash-flow projections, student-loan payoff optimization, goal tracking, deterministic financial insights, and an optional privacy-controlled OpenAI coach.

## 1. Executive summary

FinSight is implemented as a modern full-stack web application using Next.js App Router, React, TypeScript, Tailwind CSS, shadcn/Base UI components, Supabase Authentication and Postgres, and the OpenAI API. It is designed for individuals who need one place to understand balances, income, spending, recurring obligations, student loans, savings goals, and future cash-flow risk across USD, EUR, and INR.

The platform is deliberately calculation-first. Core financial results are generated by deterministic application logic. AI is an optional explanatory layer rather than the source of balances, projections, health scores, or payoff schedules.

### Current maturity

- Core product domains are implemented: authentication, accounts, transactions, CSV import, currency conversion, recurring projections, loans, goals, insights, AI coaching, settings, exports, and destructive data controls.
- The UI has undergone a broad dark-theme, responsive, and accessibility polish pass.
- Production readiness is not yet established. Automated tests, final lint/build verification, security hardening, observability, deployment runbooks, and a full regression cycle remain outstanding.

## 2. Product purpose and users

### Primary purpose

FinSight helps a user answer practical questions such as:

- What is my current financial position across all accounts and currencies?
- How much income, spending, and debt repayment occurred this month?
- What will my liquid balance look like over the next several months?
- When will my student loan be paid off at the required payment?
- How much interest and time can an extra payment save?
- Are my goals realistic relative to projected savings?
- What financial risks or unusual expenses deserve attention?
- What does my financial data mean in plain language?

### Intended users

The current application is optimized for a single authenticated individual managing personal finances. It is not currently a household collaboration product, accounting package, bank aggregation service, investment platform, tax engine, or regulated financial-advice system.

## 7. Authentication and session protection

- Authentication is provided by Supabase Auth.
- The protected dashboard layout reads claims on the server.
- Missing or invalid claims cause a redirect to `/login`.
- Authenticated user identifiers are applied to database reads and writes.
- The sidebar exposes sign-out and displays the authenticated email.
- Permanent account deletion removes the Supabase authentication user and relies on cascading ownership for related application data.
- Password-authenticated users must provide their current password before account deletion; non-password providers follow the provider-aware deletion path.

Security depends on both application query filters and correctly configured Supabase Row Level Security. RLS verification is a release requirement, not something this document certifies.

## 8. Accounts

### Capabilities

- Create an account with name, institution, type, country, currency, opening balance, and opening-balance date.
- Update account metadata.
- Activate or deactivate an account.
- Calculate current balances from opening balances and transaction history.
- Use loan accounts as debt destinations rather than ordinary transaction-source accounts.
- Prevent deletion when the account is referenced by transactions; deactivation is the safe alternative.

### Balance behavior

- Income increases the source account.
- Expense decreases the source account.
- A loan payment decreases the source cash/bank account and reduces the destination loan balance.
- Downstream pages are revalidated after account changes so overview, projections, loans, goals, and insights remain consistent.

### Important rules

- User ownership is checked for every mutation.
- Duplicate account names are handled as a database uniqueness error.
- Inactive accounts remain available for historical reporting but cannot accept new transactions in flows that require active accounts.

## 9. Transactions

### Supported transaction kinds

- Income
- Expense
- Loan payment

### Stored information

Transactions can include source account, optional destination loan, category, amount, currency, date, description, merchant or payer, notes, recurring status, recurrence frequency, recurrence start date, recurrence end date, import source, and duplicate-detection hash.

### Validation and integrity

- Source accounts must belong to the user.
- New transactions require an active source account.
- Loan accounts cannot be used as ordinary source accounts.
- Income transactions require income categories; expenses and loan payments require expense categories.
- Loan payments require a destination account of type loan.
- Source and loan destination currencies must currently match.
- Cross-currency loan payments are not yet supported.
- Future dates and invalid values are rejected by the relevant validation path.

### Recurrence

Recurring metadata is stored on the transaction. Recurring income, expenses, and loan payments become the source schedules for the projection engine.

## 10. CSV transaction import

### Workflow

1. The browser parses and normalizes the CSV.
2. The user selects the destination account and fallback income/expense categories.
3. The client sends a preview request.
4. The server revalidates authentication, ownership, account eligibility, categories, dates, descriptions, amounts, and transaction kinds.
5. The server calculates canonical SHA-256 row hashes.
6. Existing and within-file duplicates are marked.
7. The user reviews ready, duplicate, and invalid rows.
8. A commit request inserts only ready rows and records import history.

### Limits

- Maximum file size: 5 MB
- Maximum rows per import request: 500
- Imports are accepted only for active non-loan accounts
- Imported records are income or expense transactions; loan-payment import is not currently part of this flow

### Duplicate detection

The canonical hash includes account, date, kind, amount rounded to two decimals, normalized description, and normalized merchant. Duplicate checks are performed against existing transaction hashes and rows already seen in the same file.

### Category mapping

Category names are matched case-insensitively through normalized keys. Unknown categories generate warnings and fall back to the user-selected income or expense category. Missing valid fallbacks block the import.

### Import history

The `csv_imports` record stores filename, file size, total rows, imported rows, duplicates, invalid rows, status, and any error message. A successful transaction insert can still return a warning when the history record fails to save.

## 11. Multi-currency support

FinSight currently supports USD, EUR, and INR as application currencies.

- Each account and transaction retains its native currency.
- A user-selected default display currency is used for unified totals.
- Conversion uses an exchange-rate snapshot and exposes the effective rate date.
- Native amounts remain visible where context matters.
- Projections and analytics convert source values into the display currency before aggregation.
- Future foreign-exchange movement is not forecast; projections hold the latest available reference rates constant.
- Cross-currency loan payments remain unsupported.

## 12. Projections

The projection engine turns current liquid balances and recurring transactions into a month-by-month cash-flow forecast.

### Inputs

- Opening liquid balance
- Recurring income
- Recurring expenses
- Recurring loan payments
- Forecast horizon in months
- Additional monthly income
- Additional monthly expenses
- Annual income-growth percentage
- Annual expense-inflation percentage

### Monthly outputs

- Opening balance
- Income
- Expenses
- Loan payments
- Total outflow
- Net cash flow
- Closing balance

### Summary outputs

- Projected ending balance
- Total income
- Total expenses
- Total loan payments
- Average monthly savings
- Lowest projected balance
- First projected negative-balance month

### User experience

- Assumption controls support scenario exploration and reset.
- A line chart compares closing balance and monthly net cash flow.
- A table exposes all monthly calculations.
- The recurring schedule list explains the transactions powering the forecast.
- A warning appears when the balance is projected to fall below zero.

## 13. Student-loan optimization

### Loan profile

A loan account can be enriched with lender, original principal, annual interest rate, required monthly payment, next payment date, original term, and fixed or variable rate type.

### Calculation modes

- Baseline: required monthly payment only
- Accelerated: required payment plus an additional recurring amount and/or one-time payment

### Outputs

- Payoff status
- Months to payoff
- Payoff date
- Total future interest
- Total amount paid
- Principal paid
- Interest saved versus baseline
- Months saved versus baseline
- Full amortization rows: opening balance, interest, scheduled payment, extra payment, principal, total payment, closing balance

### Safeguards and limitations

- The engine flags a scenario that does not amortize within 600 months.
- Chart data is capped for display while the underlying schedule may be longer.
- The estimate uses monthly interest based on the entered annual rate.
- It does not reproduce every lender-specific daily-interest convention, fee, rate reset, capitalization rule, rounding method, or prepayment restriction.

## 14. Financial goals

### Goal types

- Savings
- Emergency fund
- Loan payoff
- Custom goal

### Progress sources

- Manual progress
- Linked-account progress

### Goal data

A goal can include target amount, current manual amount, currency, target date, linked account, linked-account baseline, planned monthly contribution, emergency-fund month target, priority, status, and notes.

### Evaluation

The goal engine calculates effective target, current progress, remaining amount, percentage complete, required monthly contribution, projected completion date, and tracking status. Tracking statuses include completed, on track, behind, no plan, no target date, and archived.

### Portfolio view

The goals dashboard aggregates total targets, total progress, planned monthly allocations, projected monthly savings, and the amount available after goal allocations. It warns when planned allocations exceed forecast savings.

### Lifecycle

Goals can be created, updated, archived, reactivated, and deleted. Linked-account baselines preserve the meaning of progress when a goal starts from an existing balance.

## 15. Insights and financial health

FinSight separates deterministic analytics from generative AI.

### Financial health score

The score ranges from 0 to 100 and is assembled from:

- Savings rate
- Emergency-fund coverage
- Debt-payment burden
- Goal health
- Cash-flow risk

The output includes a grade and component-level explanations.

### Deterministic recommendations

Insights have severity (`critical`, `warning`, `positive`, or `info`) and category (`cash_flow`, `spending`, `emergency_fund`, `debt`, `goals`, or `data_quality`). Each may include a metric and an action link.

### Spending trend

The implementation compares the last completed month with the average of the previous three completed months and reports the change percentage and top category.

### Anomaly detection

Recent expenses are normalized to the display currency. With sufficient data, FinSight applies median and median absolute deviation logic and flags unusually large transactions using a robust z-score threshold. A fallback multiple-of-median rule applies when dispersion is zero.

### Loan opportunity

For configured loans, FinSight evaluates the impact of an additional payment equal to a fraction of the required payment and surfaces the strongest interest-saving opportunity after currency conversion.

## 16. Ask FinSight AI coach

The AI coach is optional and only enabled when:

- The user enables AI features.
- AI context mode is `aggregated`.
- `OPENAI_API_KEY` is configured.
- `OPENAI_MODEL` is configured.

### Context policy

The interface states that the following may be sent:

- Aggregated balances
- Monthly income and outflow
- Savings and debt ratios
- Goal and projection summaries
- Calculated loan opportunity

The following are explicitly excluded by the current product contract:

- Passwords and API keys
- Bank credentials
- Full account numbers
- Transaction notes
- Merchant names

### Conversation behavior

- The browser sends the latest conversation window to `/api/insights/chat`.
- The UI keeps up to 12 recent messages.
- User questions are limited to 2,000 characters.
- The response may include the number of requests remaining.
- The context is read-only; the AI does not mutate financial records.
- Usage events record model, status, token counts when available, character counts, error code, and timestamp.

### Safety position

The AI explains calculations but can be incorrect or incomplete. It is not a substitute for professional financial, legal, tax, or investment advice.

## 17. Settings, privacy, export, and deletion

### User settings

- Full name
- Default currency
- Timezone
- Date format
- AI enabled/disabled
- AI context mode

Supported timezone choices currently include UTC, India, Dublin, and four continental US zones. Supported date formats include month-name, day-first, US month-first, and ISO styles.

### AI usage visibility

Settings expose recent AI usage, requests in the last 24 hours, and the daily application limit. Failed requests are excluded from the application limit according to the UI contract.

### Data export

- Complete JSON export: preferences, accounts, transactions, loans, goals, and AI usage metadata
- Transaction CSV export: spreadsheet-compatible transaction data

### Destructive controls

- Delete all transactions while preserving accounts, loan profiles, and goals
- Reset financial data by deleting goals, loan profiles, transactions, and accounts while preserving authentication, preferences, and AI usage history
- Permanently delete the account and associated rows

Confirmation phrases and password reauthentication are used for high-risk operations.

## 20. Validation and error handling

- Zod validates IDs, form inputs, and route payloads.
- Server actions return structured action states with status, message, and field errors.
- Database errors are mapped into user-facing messages for common uniqueness, relationship, and check-constraint cases.
- Missing ownership results in not-found or access-denied messaging.
- Route handlers return appropriate JSON status codes for authentication, invalid requests, missing resources, unprocessable imports, and server failures.
- Pages have loading, empty, and error states.
- High-risk actions require explicit confirmation.
- Successful mutations revalidate all downstream routes whose calculations depend on the changed data.

## 22. Financial calculation behavior

### Account balances

Balances are derived from opening balance plus/minus transactions rather than stored as an independently editable current balance.

### Currency conversion

Native values are converted through a rate snapshot into the selected display currency. Aggregations round values at presentation and financial-boundary points.

### Projection model

The model applies recurring schedules across a selected horizon and layers user assumptions for additional income, expenses, annual income growth, and inflation.

### Loan model

The amortization engine applies a monthly periodic interest rate, then allocates payments between interest and principal. Extra monthly payments and one-time payments accelerate principal reduction.

### Goal model

Goal progress can come from manual values or linked-account deltas. Target dates and planned contributions determine required contribution and projected completion status.

### Insight model

Health and recommendation logic is deterministic. The AI coach receives a summarized snapshot and explains it; it does not replace the calculation engines.

## Appendix B — terminology

- **Display currency:** Currency used to present unified totals without changing native stored amounts.
- **FX snapshot:** Effective set of exchange rates used for conversions.
- **Recurring item:** A transaction schedule used to generate projection events.
- **Baseline loan scenario:** Required payment only.
- **Accelerated scenario:** Baseline payment plus extra monthly or one-time principal payment.
- **Linked goal:** Goal whose progress is derived from an account balance.
- **Deterministic insight:** Recommendation generated by fixed application logic rather than a language model.
- **Aggregated AI context:** Summarized metrics that exclude transaction notes, merchant names, credentials, and account numbers.
